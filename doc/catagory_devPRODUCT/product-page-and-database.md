# Product Page and Product Database

## Overview

The product feature provides one public route with three modes and a dedicated product database. Category-specific style JSON controls which reusable product components appear; the database stores actual product values.

```text
/product?mode=<view|edit|new>&mainCategoryId=<main>&subcategoryId=<sub>&productId=<id>
```

## URL contracts

### Create

```text
/product?mode=new&mainCategoryId=1&subcategoryId=13
```

`mainCategoryId` and `subcategoryId` are required. A logged-in client session is required.

### View

```text
/product?mode=view&productId=<uuid>&mainCategoryId=1&subcategoryId=13
```

`productId` is required. Category IDs are ultimately read from the stored product record, so the database remains authoritative if query values disagree.

### Edit

```text
/product?mode=edit&productId=<uuid>&mainCategoryId=1&subcategoryId=13
```

`productId` and a logged-in client session are required. The record owner UID must equal the current client-session UID.

Unknown `mode` values fall back to `view`.

## Style configuration

After resolving the category pair, the page loads:

```text
/product/style/<mainCategoryId>__<subcategoryId>.json
```

The style file is generated by `/dev/category-selector`. It controls:

- Component visibility.
- Component order.
- Enabled fields within each component.
- Image count.
- Rating type.
- Order/action buttons.

It does not contain product values. Missing or invalid style files prevent the product form from being constructed and produce a visible load error.

## Rendering modes

### View

- Product data is loaded by `productId`.
- Fields are rendered as read-only values.
- Uploaded product images are shown.
- Enabled action buttons are shown.
- The page is publicly readable for active products.

### Edit

- Existing values are loaded into editable fields.
- Images use the existing storage manager and `product-default` storage profile.
- Saving calls the product update API.
- The service rejects an owner UID mismatch.

### New

- Fields and images begin empty.
- The category pair comes from the URL.
- Saving creates a UUID product record and redirects to its View URL.

## Component mapping

Visible components are filtered and numerically sorted by the style file's `order` value.

| Style key | Stored field prefix | Content |
| --- | --- | --- |
| `images` | Stored in `data.images` | Product images. |
| `rating` | `rating.*` | Rating and optional comment. |
| `price` | `price.*` | Current price, old price, needs-car value. |
| `order` | None | Cart, favorite, and contact actions from style configuration. |
| `mainData` | `mainData.*` | Name, brand, manufacturer, availability, description. |
| `specifications` | `specifications.*` | General product specifications. |
| `vehicleSpecs` | `vehicleSpecs.*` | Vehicle-specific values. |
| `propertySpecs` | `propertySpecs.*` | Property-specific values. |
| `pharmacySpecs` | `pharmacySpecs.*` | Pharmacy-specific values. |

The central `fields` object uses stable dotted keys such as `mainData.name` and `propertySpecs.rooms`. This keeps the database schema stable when category-specific fields change.

## Image storage

Editable modes use `StorageImageManager` with `StorageProfiles.ProductDefault`.

The product page does not select a provider, folder, output format, or size
limit. It supplies only the profile ID, as required by the mandatory image
storage architecture contract in `doc/system/image system`.

The complete runtime pipeline is:

```text
ProductPageContent
  -> StorageImageManager
  -> useStorageProfileUpload
  -> ImageStorageService
  -> ImageStorageApiService
  -> POST /api/storage/images/upload
  -> ImageUploadApplicationService
  -> ImageStorageOrchestrator
  -> Storage profile
  -> Runtime provider
```

`src/config/storage-profiles.json` defines `product-default` with folder
`images/products/default`, WebP output, and a 30 KB processed-image limit.

Provider selection is environment-driven inside
`provider-resolver.server.ts`:

- Development (`NODE_ENV=development`) always selects
  `LocalStorageProvider`, regardless of the provider declared in the profile.
  Product files are written under
  `public/sync_data/sync_file/images/products/default/<uuid>.webp` and their
  public URL starts with `/sync_data/sync_file/images/products/default/`.
- Production, static, and Capacitor server targets use the provider declared
  by the profile. `product-default` declares `CloudflareR2`, so product images
  are uploaded to R2 under `images/products/default/<uuid>.webp` and receive an
  R2 public URL.

- Files are compressed according to the product storage profile.
- Uploads use the existing `/api/storage/images/upload` pipeline.
- The product record stores the resulting `imageKey` and resolved URL.
- Maximum images comes from the category-pair style file.
- View mode reads stored image references and does not expose upload controls.

## Product image gallery

The real `/product` page uses `ProductImageGallery` in View mode.

- The active image is rendered as one large main image.
- With one image, no thumbnail row is rendered.
- With multiple images, every thumbnail appears in a horizontally scrollable
  row below the main image and the active thumbnail has a primary border.
- With no images, one unified placeholder is rendered.
- Thumbnail taps update the active image and reset zoom/pan.
- Touch swipe changes the active image.
- Touch pinch zooms from 1x to 4x.
- Double tap toggles zoom, and a zoomed image supports touch pan.
- Mouse-specific gallery interaction, arrows, zoom controls, hover behavior,
  magnifier, and fullscreen are intentionally absent.
- The initial main image is eager/high-priority. Thumbnails use native lazy
  loading and asynchronous decoding. Previously loaded URLs are tracked for
  smooth opacity transitions and the browser cache prevents repeat downloads.

In Edit and New modes, `ProductImageEditors` renders the style-configured
number of independent `StorageImageManager` instances. Every instance owns one
slot (`maxItems: 1`) and uses `StorageProfiles.ProductDefault`.

## Client implementation

- Route parser: `src/app/product/page.tsx`
- Page UI: `src/components/product/ProductPageContent.tsx`
- Client API adapter: `src/features/product/services/product-api-service.ts`
- Shared product entities: `src/features/product/entities/product.entity.ts`

The server page normalizes URL mode and passes plain values to the client page. The client page reads the IndexedDB-backed session through `SessionProvider`, loads the product/style data, renders components, and submits create/update requests.

## Business API

Endpoint:

```text
/api/products
```

Canonical route constant: `GOVA_API_ROUTES.products`.

### GET

```text
GET /api/products?id=<productId>
```

Returns an active or draft product. Archived records are treated as not found.

### POST

Creates a product from:

```json
{
  "uid": "owner-uid",
  "mainCategoryId": "1",
  "subcategoryId": "13",
  "data": {
    "fields": { "mainData.name": "Example" },
    "images": []
  },
  "status": "active"
}
```

The service generates the UUID and timestamps.

### PUT

Updates mutable data/status. The submitted UID must equal the stored owner UID. Category IDs and ownership cannot be changed by this endpoint.

## Validation

- Product/category identifiers accept alphanumeric characters and hyphens.
- Field keys accept letters, digits, underscore, dot, and hyphen.
- Field values must be strings and are truncated to 10,000 characters.
- Images require string `imageKey` and `url` values and are limited to 20 at the service boundary.
- Status is normalized to `active`, `draft`, or `archived`.
- Archived products are not publicly returned.

## Database architecture

The feature uses a third database independent from `allusers.db` and `profile.db`.

### Development

```text
public/sync_data/sync_sqlite/product.db
```

### Production

Turso credentials:

```text
TURSO_PRODUCT_DATABASE_URL
TURSO_PRODUCT_AUTH_TOKEN
```

The product database follows the existing environment-based driver rule:

- Development uses `ProductSQLiteDatabaseClient`.
- Production uses `ProductTursoDatabaseClient`.

Main database files:

- `src/core/database/product-db-client.ts`
- `src/core/database/product-sqlite-db-client.ts`
- `src/core/database/product-turso-db-client.ts`
- `src/core/database/product/product.schema.ts`

## Products table

```sql
CREATE TABLE products (
  id TEXT PRIMARY KEY NOT NULL,
  uid TEXT NOT NULL,
  main_category_id TEXT NOT NULL,
  subcategory_id TEXT NOT NULL,
  data_json TEXT NOT NULL DEFAULT '{}',
  status TEXT NOT NULL DEFAULT 'active',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

Indexes:

- `products_uid_idx` on `uid`.
- `products_category_idx` on `(main_category_id, subcategory_id)`.

`uid` is intentionally not a cross-database foreign key because the owner exists in the separate users database.

### `data_json`

```json
{
  "fields": {
    "mainData.name": "Example product",
    "price.current": "1250",
    "propertySpecs.rooms": "3"
  },
  "images": [
    {
      "imageKey": "stored-object-key",
      "url": "https://storage.example/product.webp"
    }
  ]
}
```

## Database creation and migrations

Create the local database explicitly:

```text
npm run db:create:product
```

Ensure all three local databases:

```text
npm run db:ensure
```

The product SQLite client also applies migrations lazily on first access in development. Drizzle configuration is in `drizzle.product.config.ts`.

Production Turso must have the SQL migration applied before serving product API traffic.

## Repository and service layers

- Repository: `src/features/product/repositories/product-repository.ts`
- Server service: `src/features/product/services/product-service.server.ts`
- API route: `src/app/api/products/route.ts`

SQL exists only in the repository. The service owns normalization, generated identifiers, timestamps, status rules, and ownership checks. The API maps domain errors to HTTP status codes.

## Authentication boundary

The current application session is stored client-side in IndexedDB. There is no cryptographically authenticated server session/token attached to product API requests.

Current protection therefore consists of:

1. The UI blocks New/Edit when no client session exists.
2. Update verifies that the submitted UID equals the stored owner UID.

This prevents accidental cross-owner editing through the normal UI but is not sufficient against a caller who forges another UID. Before exposing write operations to untrusted production clients, add authenticated server-issued session credentials and derive UID on the server rather than accepting it from request JSON.
